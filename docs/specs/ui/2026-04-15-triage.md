## 1. Jobs-to-be-done

All four agent shapes are valid first-class use cases the production UI must serve:

- **Proactive team agent** (code-guardian archetype) — schedules, memory, channel output
- **Personal assistant** — interactive chat, per-session work
- **Autonomous worker** — scheduled runs, workspace artifacts
- **Conversational channel bot** — primarily Slack/Telegram, minimal web presence

**Core platform features** driving UI priorities: scheduled jobs and chat (web UI + external channels). No heavy dashboards in the PoC.

**Ranked reasons a user opens the UI** (most → least common):

1. *Scheduled agent ran — review what it did*
2. *Something broke — debug*
3. *Chat with an agent ad-hoc*
4. *Saw something in Slack — open web for more context*
5. *Change behavior — update rules, connectors, schedule*
6. *Create a new agent*

The #1 reason is retrospective (reviewing autonomous work), not interactive. This shapes the home surface.

## 2. North stars (binding constraints)

Derived from existing ADRs and resolved during the grill:

- **Chat-primary, dashboard for inspection** (ADR-013)
- **Sessions ARE the activity log** — no separate "activity feed" concept. Every schedule run posts into its own session; the sorted session list is the timeline.
- **Single-user PoC persona**; multi-user RBAC post-PoC (ADR-015)
- **Web UI is owner-only**; team access happens via Slack (#105 interpretation). Each team member can own their own instance.
- **Channel sessions hidden from the main session list** (ADR-016)
- **DB-backed sessions** with `regular` / `channel_slack` discriminator (ADR-017)
- **The pre-PoC prototype is not preserved** (ADR-013). Only useful pieces (tRPC clients, ACP connection logic) carry forward.

## 3. Information architecture

```
│  [humr]  [Home] [Connectors] [Secrets]    [theme] [user ▾]   │
```

Three user-global destinations for PoC: Home, Connectors, Secrets. (API keys destination arrives with #94 post-PoC; likely triggers consolidation into an "Account" dropdown at that point.)

### 3.2 Home

One section: **Your agents**, sorted by most-recent child-instance activity.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Your agents                             [+ New Agent]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

▾ code-guardian                    [+ Instance] [···]
    ● acme-repo      · Heartbeat · 7min ago  · 3 new   [Open]
    ○ private-repo   · hibernated · yesterday          [Open]

▾ claude-code                      [+ Instance] [···]
    ● exp            · My Chat    · 2h ago             [Open]

▾ research-bot                     [+ Instance] [···]
    ⚠ main           · ERROR      · 4h ago   · 1 new   [Open]
```

**Rules:**

- Agents sorted by `max(child instance lastActivityAt)` desc
- Instances within an agent sorted by own `lastActivityAt` desc
- **Errored instances float to the top within their agent group** regardless of timestamp
- Per-instance row: state dot, name, latest session type/name, relative timestamp, unread badge, Open action
- Agents expanded by default for PoC
- Empty states: (no agents) big centered CTA; (agent with no instances) placeholder row

For schedule-backed sessions (heartbeat/cron/improvement), the Chat tab gains:

```
─────────────────────────────────────────────────────────
📅  Heartbeat · every 5 min
    288 runs in 24h  ·  3 errors  ·  Last: 07:15 ✓ (12 min ago)
    [◀ Previous run]   [Jump to latest ▶]
─────────────────────────────────────────────────────────

═════════════════════════════════════════════════════════
 April 15 · Tuesday
═════════════════════════════════════════════════════════

─── ✓ Run at 07:10 ─────────────────────────────────────
  [agent messages]
  [tool chips]

─── ✓ Run at 07:15 ─────────────────────────────────────
  [agent messages]

─── ⚠ Run at 07:20 · error ─────────────────────────────
  [agent messages including error]

─── ─ ─ ─ ─ ─  New since your last visit  ─ ─ ─ ─ ─ ───

─── ✓ Run at 07:25 ─────────────────────────────────────
```

**Open behavior:** jump to first-unread run using `lastVisitedAt`; fall back to latest if fully read.

**Keyboard:** `J` / `K` next/previous run; `End` jump to latest; `Home` jump to first unread.

**Run boundaries source:** explicit meta-message emitted by the trigger-watcher on each run start, falling back to a >60s timestamp-gap heuristic if absent.

### 7.4 Inline permission card

Session-origin-based policy:

| Session origin | Permission behavior |
|---|---|
| Interactive web chat (`regular`) | Prompt user inline |
| Scheduled (cron/heartbeat/improvement) | Auto-approve all |
| Channel (`channel_slack`, future Telegram) | Auto-approve all |

When prompted:

```
┌─────────────────────────────────────────────┐
│ ⚠  Agent requests permission                │
│                                             │
│  Edit file: src/App.tsx                     │
│                                             │
│  ☐ Allow this kind for this session         │
│                                             │
│  [ Deny ]    [ Allow ]                      │
└─────────────────────────────────────────────┘

[ Waiting for your decision… ]   ← input disabled
```

- Blocking — input area disabled; agent stream paused
- Buttons come from ACP protocol options (not hardcoded Allow/Deny)
- After decision, card collapses to one-line audit:
  `✓ Allowed: Edit file src/App.tsx  (you, 12:34)`
- Keyboard: Enter activates focused button
- Per-session memory via checkbox
- If user navigates away: permission persists, badge appears on Home + Sessions rail

## 9. Cross-cutting principles

**Empty states everywhere.** One-sentence explanation of what belongs here + CTA where applicable. No blank lists.

**Three-layer errors:**

| Error type | Treatment |
|---|---|
| Platform down (API unreachable) | Full-width banner, auto-retry with backoff |
| Instance-level (agent crash) | Instance state = error, inline card in chat |
| Action-level (create/delete failed) | Bottom-right toast with Retry |
| Auth expired | Redirect to Keycloak preserving return URL |

**Optimistic updates:** no. Pessimistic for PoC.

**Notifications for PoC:** badges (unread, HiTL pending) + action toasts only. No notification center, no browser notifications, no email/Slack alerts.

**Responsive:** single layouts scale down. Fix breakage on narrow widths only. No custom mobile designs in PoC.

**Accessibility:** best-practice defaults from the component library. No deep audit for PoC.

**ACP artifacts:** render image content blocks inline in messages. Future media types as they appear.
