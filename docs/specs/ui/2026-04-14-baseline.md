# UI Baseline — Pre-PoC Teardown

> Snapshot of the current UI state as of 2026-04-14. This document serves as the
> baseline for defining user stories and designing the production UI.

---

## 1. Bootstrap & Authentication

**Flow:** `main.tsx` → `initAuth()` → render `<App />` or redirect.

1. Fetch OIDC config from `GET /api/auth/config` → returns `{ issuer, clientId, onecliUrl }`
2. Create `UserManager` (oidc-client-ts) with Keycloak authority, redirect URI `/auth/callback`, sessionStorage for user store, `automaticSilentRenew: true`
3. If pathname is `/auth/callback` → call `signinRedirectCallback()`, restore saved return URL from `sessionStorage["humr-auth-return"]`, replace URL, return user
4. Otherwise check existing session via `getUser()` — if valid and not expired, return user
5. If no valid session → save current `pathname + search` to `sessionStorage["humr-auth-return"]`, redirect to Keycloak
6. If `initAuth()` returns `null` (redirect in progress), app does NOT render at all
7. If user returned → dynamically `import("./App.js")` and render in `StrictMode`

**Token management:**

- `getAccessToken()` returns cached token if valid, attempts `signinSilent()` if expired, falls back to `signinRedirect()` (throws `"Session expired"`)
- `authFetch()` wrapper injects `Authorization: Bearer <token>` header
- Both tRPC clients (`platform`, `createInstanceTrpc`) use async headers function calling `getAccessToken()`
- WebSocket URL includes token as query param: `ws://host/api/instances/:id/acp?token=TOKEN`

**Logout:** calls `userManager.signoutRedirect()` → Keycloak logout, redirects to `/`

## 2. App Shell & Navigation

**Layout:** Two layouts depending on `view` state:

- **`view === "chat"`** → full-screen `<ChatView />` + `<DialogOverlay />` (no shell/nav)
- **`view === "list"` or `"connectors"`** → shell with nav bar, blob background, max-width 960px content area

**Nav bar** (sticky, 48px tall, backdrop-blur):

- Logo: Shell icon + "humr" text in accent color
- Two tab buttons: "Agents" (`view: "list"`) and "Connectors" (`view: "connectors"`) — active tab gets accent background
- Conditional "OneCLI" external link — only shown if `getAuthConfig()?.onecliUrl` is truthy
- Theme toggle: 3-button group (Sun/Moon/Monitor icons) — active button gets accent background with white icon
- User display: shows `preferred_username` or `sub` from OIDC profile
- Logout button (LogOut icon)

**Routing (manual, no library):**

- `pathToState(path)` → parses `/chat/:id` → `{view: "chat", instance: id}`, `/connectors` → `{view: "connectors"}`, else `{view: "list"}`
- `viewToPath(view, instance)` → reverse mapping
- `setView(v)` → `history.pushState()` + set store
- `selectInstance(id)` → pushState to `/chat/:id` + reset session/messages/files/log state
- `goBack()` → pushState to `/` + reset all chat state + re-fetch agents & instances
- `popstate` listener in `App.tsx` → parses URL, resets appropriate state
- Initial URL handling on mount: if pathname starts with `/chat/`, sets state directly (no pushState to avoid duplicate)
- Auth return: `sessionStorage["humr-return-view"]` restores view after OAuth redirect (used by connectors page)

**Theme:**

- Persisted to `localStorage["humr-theme"]` — values: `"light"`, `"dark"`, `"system"`
- Applied by toggling `.dark` class on `<html>`
- System preference tracked via `matchMedia("(prefers-color-scheme: dark)")` change listener
- Both `App.tsx` effect and `store.setTheme()` apply the theme (double coverage)

**Background decoration:**

- 3 floating blob `<div>`s with blur(80px), absolute positioned, accent/orange colors at low opacity
- Each blob has its own floating animation (14s, 16s, 12s cycles)
- Dark mode reduces opacity further

**OAuth result handling:**

- On mount, checks `?oauth=error&message=...` query params
- If found, strips query params from URL and shows alert dialog with error message

**Polling on mount:**

- `fetchTemplates()` once
- `fetchAgents()` once
- `fetchInstances()` immediately + every 5 seconds via `setInterval`

## 9. ChatView — The Core Chat Interface

**Layout:** Full-screen 3-column flex with blob background:

- Left sidebar (resizable, default 220px, min 140px, max 400px)
- Center chat column (flex-1)
- Right sidebar (resizable, default 340px, min 240px, max 600px)
- ResizeHandles between columns — widths persisted to `localStorage["humr-left-w"]` and `localStorage["humr-right-w"]`

**On mount effects:**

1. Creates per-instance tRPC client via `createInstanceTrpc(selectedInstance)`
2. Resolves enabled MCP servers from agent config + instance enabled list
3. Fetches file tree once, then polls every 2 seconds:
   - Refreshes full tree
   - If a file is currently open, re-fetches its content (updates live)
   - If the open file no longer exists, clears it (sets `null`)
4. **Auto-wake:** If instance state is `"hibernated"` on entry, calls `platform.instances.wake.mutate({id})` silently
5. Fetches sessions with retry: calls `fetchSessions()`, if fails (instance not running yet), retries every 3 seconds until success
6. Auto-scrolls to bottom on every messages change

**Header bar** (44px, backdrop-blur):

- Back button: "← Agents" — closes WebSocket, clears connection ref, calls `goBack()`
- Vertical separator line
- Instance ID as title (bold, 14px)
- "New Session" button (only shown if `sessionId` is set) — closes WebSocket, clears session/messages
- State badge: shows instance state or "Busy" (with warning color/pulse) when agent is processing

**Message area:**

- Max-width 760px centered, 32px padding
- **Loading session state:** spinner + "Loading session..." (shown during `resumeSession`)
- **Empty state:** "Start a conversation" / "Send a message to begin a new session with this agent"
- **Messages rendered** as a vertical list with 24px gap:
  - Each message: "You" or "Agent" label (tiny uppercase muted), then content
  - **User messages:** rounded bubble with accent-light background, accent border, right-aligned, max-width 620px, whitespace-pre-wrap
  - **Assistant messages:** left-aligned, full-width, parts rendered sequentially:
    - Text parts: rendered as `<Markdown>` with file link support
    - Tool parts: rendered as `<ToolChip>` components
  - **Streaming indicator:** blinking orange cursor (7px x 16px block, 1s blink animation):
    - On user message: appended after last text part
    - On assistant message: shown as standalone block if no parts yet

**Input area** (border-top, backdrop-blur, 16px padding):

- Textarea: max-width 760px, auto-height (min 44px, max 180px), border glow on focus
- Enter sends, Shift+Enter for newlines
- Disabled during `busy` state
- Send button: accent color with brutalist shadow, shows "..." when busy, disabled when busy or empty input

## 10. ACP WebSocket Protocol — Message Sending Flow

**`send()` function — the core interaction loop:**

1. Trim input, bail if empty/busy/no instance
2. Clear input, set `busy: true`
3. Create user message + empty streaming assistant message, append to messages
4. Log "prompt" event
5. **Connection management:**
   - If no connection or WebSocket closed → call `openConnection()`:
     - Build WS URL with auth token as query param
     - Create WebSocket with 120s connect timeout
     - Wrap in ReadableStream/WritableStream pair
     - Create `ClientSideConnection` with handlers:
       - `requestPermission` → **auto-accepts first option** (no user prompt)
       - `sessionUpdate` → delegates to `onUpdate` callback
       - `writeTextFile` / `readTextFile` → no-ops (return empty)
     - Call `connection.initialize()` with protocol version, fs capabilities
     - Set `ws.onclose` to null out connection ref
     - Store in `connectionRef`
6. **Session management:**
   - If no `activeSessionIdRef` (first message):
     - If `sessionId` exists in store (resumed session) → `unstable_resumeSession({ sessionId, cwd: ".", mcpServers })`
     - Else → `newSession({ cwd: ".", mcpServers })` → receive `sessionId` → set in store, log "session" event, register with API via `platform.sessions.create.mutate()` (fire-and-forget, catch ignored)
7. **Send prompt:** `connection.prompt({ sessionId, prompt: [{type: "text", text}] })`
8. **On completion:** Mark assistant message as `streaming: false`, log "done" event with `stopReason`
9. **On error:** Log "error" event, replace assistant message parts with error text, close WebSocket, null out refs
10. **Finally:** `setBusy(false)`, re-fetch sessions, focus textarea

**Update handler (streaming):**

- `agent_message_chunk` (text type): Append text to current assistant message's last text part (or create new text part). Also logs "text" event.
- `tool_call`: Append ToolChip to assistant message parts. Logs "tool" event.
- `tool_call_update`: Find matching ToolChip by `toolCallId` across all parts, update status/title/content.
- **Text stripping:** Both user and agent messages strip XML-like tags: `/<[a-z-]+>[\s\S]*?<\/[a-z-]+>/g` → removes internal protocol tags from display

## 11. Session Resume Flow

**`resumeSession(sid)` — loads full history of an existing session:**

1. Set busy, loading, clear messages, set sessionId
2. Close any existing WebSocket
3. Create fresh connection with update handler that builds messages into a `Map<messageId, Message>` + ordered `string[]`
4. Key grouping logic for assistant messages: finds last user message in order, then looks for existing assistant message after it — creates one if none exists. This groups all agent responses between user prompts into a single assistant message.
5. Update handling during load:
   - `user_message_chunk`: Create/append user message by `messageId`
   - `agent_message_chunk`: Append to current assistant message (after last user)
   - `tool_call`: Append chip to current assistant message
   - `tool_call_update`: Find chip by `toolCallId` across all messages, update
6. After `connection.initialize()` + `connection.loadSession({ sessionId, cwd: ".", mcpServers })` → WebSocket is closed
7. Convert Map to ordered Message array, set as messages
8. Clear loading/busy state

## 13. FilesPanel — File Explorer & Viewer

**Two modes:** file tree or file viewer (when `openFile` is set)

**File tree mode:**

- Flat list from `fileTree: TreeEntry[]`, sorted: dirs before files at each depth level, alphabetical within type
- Each entry indented by depth (14px per level + 12px base)
- Dirs: ChevronRight/ChevronDown toggle icon + Folder icon + name — click toggles collapse state (tracked in local `Set<string>`)
- Files: FileText icon + name — click → `onOpenFile(path)` which fetches file content and switches to right tab "files"
- Collapse logic: entry is hidden if any ancestor dir is in the collapsed set
- Empty state: "No files yet"

**File viewer mode:**

- Header bar: "← Back" button + file path (monospace, truncated) + optional render/raw toggle for markdown files
- Toggle: Eye icon "Render" ↔ Code icon "Raw" — only shown for `.md` / `.mdx` files
- Content area:
  - Markdown files with render on: `<Markdown>` component with file link support
  - Everything else: `<HighlightedCode>` with syntax highlighting
- File link behavior in markdown: relative links (not http/mailto/#) are intercepted, normalized (strip `./`), and trigger `onFileClick` → loads that file in the viewer
- **Toggle behavior:** clicking the same file path again closes the viewer (`setOpenFile(null)`)

## 16. Reusable Components

**StatusIndicator:** 10x10px colored dot. Maps instance states to colors: running=green, starting/hibernating=warning+pulse animation, hibernated=gray, error=red. Falls back to green for unknown states (including `"ready"` used by connectors).

**ToolChip:** Expandable tool call display. Shows chevron (if content exists) + status icon (Check/X/Loader-spin/nothing) + title text. Click expands to show content blocks (pre-formatted monospace text with markdown fences stripped). Status colors: pending=muted, in_progress/running=warning, completed=success, failed=danger.

**Markdown:** ReactMarkdown wrapper with GFM plugin + rehype-highlight for code blocks. Relative links (not http/mailto/#) are intercepted and passed to `onFileClick` callback. External links open in new tab.

**HighlightedCode:** highlight.js with 11 registered languages (JSON, Python, JS, TS, YAML, Bash, CSS, XML, Go, SQL, Dockerfile). Maps file extensions to languages. Falls back to plain `<pre>` if no language match. Uses `dangerouslySetInnerHTML` for highlighted output.

**DialogOverlay:** Global modal driven by store's `dialog` state (null = hidden). Alert = single OK button, Confirm = Cancel + Confirm. Keyboard: Escape → cancel, Enter → confirm. Auto-focuses confirm button on open. Click backdrop → cancel.

**ResizeHandle:** 5px wide draggable bar between panels. Tracks mouse drag via `mousemove`/`mouseup` on `document`, sets `col-resize` cursor and `user-select: none` during drag. Calls `onResize(delta)` on each move. Hover/active states with accent color.

## 17. State Management — Zustand Store

Single flat store with no slices or middleware. All state is synchronous except async action methods.

| State | Type | Default | Persistence |
|---|---|---|---|
| `dialog` | `DialogState \| null` | `null` | none |
| `theme` | `"light" \| "dark" \| "system"` | from localStorage or `"system"` | `localStorage["humr-theme"]` |
| `view` | `"list" \| "chat" \| "connectors"` | from URL or sessionStorage | URL (pushState) |
| `availableChannels` | `Record<string, boolean>` | `{}` | none |
| `templates` | `TemplateView[]` | `[]` | none |
| `agents` | `AgentView[]` | `[]` | none |
| `instances` | `InstanceView[]` | `[]` | none |
| `selectedInstance` | `string \| null` | `null` | URL path |
| `sessions` | `SessionView[]` | `[]` | none |
| `messages` | `Message[]` | `[]` | none |
| `schedules` | `Schedule[]` | `[]` | none |
| `fileTree` | `TreeEntry[]` | `[]` | none |
| `openFile` | `{path, content} \| null` | `null` | none |
| `log` | `LogEntry[]` | `[]` | none |
| `busy` | `boolean` | `false` | none |
| `sessionId` | `string \| null` | `null` | none |
| `rightTab` | `"files" \| "log" \| "schedules"` | `"files"` | none |
| `includeChannelSessions` | `boolean` | `false` | none |
| `loading` | `{templates, agents, instances, sessions, session}` | all `false` | none |

**Error handling pattern:** All async actions use try/catch. Create/delete/connect operations show alert on error via `showAlert()`. List/fetch operations silently swallow errors (empty catch blocks).

## 20. Known PoC Limitations & Technical Debt

1. **No router library** — manual pushState/popstate with string parsing; no route guards, no 404 handling, no nested routes
2. **Heavy polling** — 2s file tree + 5s instances + 5s schedules + 3s session retry; no Server-Sent Events or WebSocket push for state changes
3. **Flat Zustand store** — 30+ fields in one object, no slices/selectors optimization
4. **Dead code** — `CreateTemplateDialog` exists but is never imported or rendered
5. **CreateTemplateDialog uses `fetch` without auth** — unlike `AddAgentDialog` which uses `authFetch`
6. **Permission auto-accept** — `requestPermission` always selects first option, never prompts user
7. **No error boundaries** — React errors will crash the entire app
8. **No loading skeletons** — raw boolean flags, some "Loading..." text
9. **No pagination** — all lists fetch everything
10. **Slack is "paste a bot token"** — no OAuth flow, no validation, no feedback on success
11. **File viewer is read-only** — no editing, no search, no breadcrumbs, no line numbers
12. **No textarea auto-resize** — fixed min/max heights, `rows={1}` with manual overflow
13. **Instance ID shown as title** — not instance name
14. **Session title is `title || sessionId.slice(0,12)`** — often just a truncated UUID
15. **Message rendering duplicates update logic** — `send()` and `resumeSession()` each implement their own update handling with different patterns (immutable state updates vs. mutable Map)
16. **No WebSocket reconnect** — if connection drops during a session, user must send a new message to reconnect
17. **No confirmation before navigating away** — leaving chat during streaming doesn't warn
18. **Connectors state is local** — not in Zustand store, can't be referenced from other views without re-fetching
19. **No accessibility** — no ARIA roles, no skip navigation, no screen reader support
20. **No responsive/mobile support** — fixed 3-column layout, no breakpoints
